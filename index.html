Task-X

Overview

Task-X is a twist on a todo app that takes users through a social media esque journey. It uses the Google sign-in API with PassportJS, integrating with MongoDB through the use of MongooseJS. The database holds user profiles, users' tasks, amount of likes and comments on tasks, and notification data. The backend is Node with ExpressJS and the frontend is React with Redux for state management.

Upon signing in with Google for the first time users are taken through a profile setup flow where they can choose an avatar and a username. While the user types their potential username an event is debounced that checks against the database to ensure the username is available. I'm not trained in UI/UX but I used Framer (a great animation library) and Tailwind to give it a decent style.

Once signed in a user can create tasks with a due date and are given the ability to add the task to the Task Wall. The Task Wall is where users allow other users to interact with their task. Users can comment and like tasks. After a task receives a certain amount of likes these tasks gain awards. Users are notified of awards, comments and likes which funnel down from the database to be presented on the frontend. Finally, users can delete their accounts which removes all posts, likes and comments from the database.

Doublemap
mapbox version leaflet version

Overview

The Mapbox version of Doublemap uses JQuery with a PHP backend and MySQL for database needs. Tailwind was used to give the application a decent feel and design. It was deployed using Docker on a VPS with an Nginx container and a PHP-FPM container. The map has a modern side and a historical side.
The modern side delivers:
  • Points of interest based on where the map is centered using the OpenStreetMap Overpass API. The user can expand the search for POIs at will.
  • A search box that uses the Mapbox Search API to deliver countries, regions, cities, towns and streets.
  • Users can select countries with the cursor or the select box and get information on a countries population, capital, region etc.
The historical side delivers:
  • A range slider allowing users to choose a day of the year where the Wikipedia 'On This Day' API is used and combined with ChatGPT API to convert events to latitude and longitude that can be mapped.
  • OpenHistoricalMap API is used so when users choose an event in time the tiles on the map change to represent the year in question, giving a context to the event.
  • Users can select countries with the cursor or the select box where the history of that country is retrieved from Wikipedia and presented to the user.

Challenges and Solutions

There were many challenges and things I had to experiment with while creating this map but I will name the most significant:
  • Challenge 1: Is there an API that provides historical events with latitude and longitude? If not, how can this be done and what possible solutions are there?
  • Challenge 2: ChatGPT API costs and can be slow in response especially GPT-4. GPT-3 is not accurate and can't be trusted, how can I get this working?
  • Challenge 3: Mapbox also costs after a certain amount of requests, how can I ensure I don't go over this limit?

Solution 1

After a few weeks of research I realised my needs were very niche and no API would provide what I needed. I knew Wikipedia would be a great resource but didn't know where to start. After researching the API I found 'On This Day'. Events that stretch back sometimes 2000 years - perfect! Wait, these events don't come with latitude and longitude...

On inspection the events have text with popular figures in pop culture, famous landmarks, and many times country, city or town names. There must be a way to deduce coordinates in this modern age if I have all this information to hand, right? Enter ChatGPT.

The answer was clear. ChatGPT excelled at this. Within these events are famous places, historical landmarks, and well-known figures. With the right approach, there had to be a way to extract locations. ChatGPT could recognize names and infer places even when they weren't explicitly mentioned. It wasn't just matching text - it was reasoning, connecting history, and filling in the gaps where a traditional API would fail. It could adapt, making it the perfect tool to bridge the missing latitude and longitude gap.

Solution 2

ChatGPT costs. After testing, GPT-3 is less expensive at the cost of reasoning and accuracy, struggling with ambiguous locations or missing context. It might recognize a famous landmark but fail entirely to provide its coordinates, even mixing up events (as they are sent in batches). GPT-4, on the other hand, excels with a better understanding of historical events, better disambiguation, and a good grasp of geographic clues. GPT-4 is much slower and costs more with much better results, while GPT-3 is much faster and cheaper with less accurate results. Solution?

I had to use GPT-4 otherwise this was pointless. Streaming the response to my PHP backend was the key to prevent timeouts. How to reduce costs overall? Storing the events was the key, that way a user makes one request for that specific day, I could be almost certain I wouldn't have to send the events to GPT ever again. Enter MySQL. After receiving the events back from ChatGPT I would fill my database up with the events. When a user selects the day again, the events would be ready, and if ChatGPT failed to provide coords for one of the events, I would send a batch of these events again on this new request. This worked surprisingly well!

Solution 3

This one is similar to the second problem and was a more easy solution. Loading the map and searching POIs go towards two seperate API calls that will cost after a certain amount of requests each month. Once again nothing a simple MySQL table can't solve.

I setup a table to track the amount of requests per API and increment the counts after each request. If the app is at the limit the request never makes it to Mapbox and a 429 error is returned to the user letting them know no more requests can be made. To reset the request counts on the 1st of the month I created a scheduled event. Viola, problem solved.

The leaflet version of Doublemap, Gazeteer, uses JQuery and PHP to perform routines retrieving news articles, wikipedia articles, country information, weather for the capital and providing a currency converter. Tailwind and Bootstrap give the app a modern and simple design. Docker was used for deployment on a VPS with an Nginx container and a PHP-FPM container.

When a user first uses the map they are asked to give the map permission to access location using the Geolocation API. This is where the routine finds the coordinates and can select the correct country and query PHP to retrieve the country's information. There are also cities and airports pinned to the map and clustered.

Whatsong

Overview

Whatsong integrates with Spotify API to provide users the option to search songs and artists. It is a React/Typescript app that makes use of reducers and the useContext hook to handle the most important data in the application. React Router is used to route the user between the components. Framer Motion is used to give the app a decent design experience.

Jest, React Testing Library and Mock Service Worker are used to test the user interface and network requests to the Spotify API. Axios is used to make all network requests. Users choose artists or songs and can search either one. The artists side gives users detailed information on that artist including top tracks and album lists. The songs sides lets users grab an element and drag it into a highlighted box in order to see the song's details.

Looking back on Whatsong there are many things I would love to upgrade about this including the design and the knowledge I now have on React Router. React Router could have been implemented better. I also eventually want to move this over to Docker, add a Laravel backend, create an auth flow and allow users to save tracks, artists, songs and albums as favorites - all integrated with a user dashboard.

Company Directory

Overview

A directory that gives the user control over the company's records. Users manipulate records with a search input, filters for personnel, edit and delete buttons for records and also the ability to add records with an add button. A JQuery frontend directs these network requests to the PHP backend, where the MySQL database is queried. It was deployed using Docker on a VPS with an Nginx container and a PHP-FPM container. Tailwind and Bootstrap are used to give a simple and effective UI.

The MySQL database is the star of the show ensuring that records with dependencies aren't deleted with defined foreign keys and delete restrictions as well as cascading updates to those related records. The PHP backend has extensive error handling to explain these things to the user in a user-friendly way. Personnel record filters allow users to search by department or location. A query param is added the request URL with the query and the filter, which is added as a WHERE clause to the mySQL query on the backend.
